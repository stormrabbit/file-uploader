## Context

`UploadService.uploadAssets`（`@d:\ubw_personal\file-uploader-v2\file-uploader-app\lib\services\upload_service.dart:164-225`）串行处理一批 `AssetEntity`。每张资源的处理流程为：读取原始文件 → 计算 MD5 → 秒传查询（`isExist`）→ 若未命中则 `uploadFile` 实际上传。当前任一步骤抛出异常，直接进入 `catch`，记录日志并调用 `onItemDone(asset.id, null)`，跳到下一张。

网络抖动、服务端瞬时 5xx、超时等是常见的可恢复失败原因，值得在放弃前做有限次数的自动重试。要求改动最小，不引入新依赖、不改变对外签名。

## Goals / Non-Goals

**Goals:**
- 在单张资源的 `try/catch` 内部实现「失败自动重试 3 次（共至多 4 次尝试）+ 指数退避」。
- 全部重试用尽后才执行现有失败路径（`onItemDone(asset.id, null)` + 日志）。
- 不改变 `uploadAssets` 方法签名、`onProgress`/`onItemDone` 回调契约、多资源之间的串行顺序。

**Non-Goals:**
- 不引入可配置的重试策略（次数/退避基数硬编码为最小实现，符合“改动最小”的要求）。
- 不区分错误类型做选择性重试（不做「哪些异常可重试」的精细分类），所有异常统一按同一重试策略处理。
- 不改变秒传命中路径（`isExist` 命中分支不需要重试上传本身）。
- 不新增单测框架或 mock 基础设施，除非现有测试目录已具备。

## Decisions

1. **重试范围**：将单张资源处理中「MD5 计算 → 秒传判断 → uploadFile」整体作为一次尝试单元，包裹在一个内部循环里，而不是单独重试 `uploadFile`。
   - 理由：改动最小；异常可能来自 MD5/秒传查询任一环节，统一重试逻辑简单可靠，无需拆分成多个可重试子步骤。
   - 备选方案：只重试 `uploadFile`（MD5/秒传失败不重试）。放弃原因：需要额外分支处理，增加复杂度，且秒传查询失败同样值得重试。

2. **重试次数与退避**：最多重试 3 次（初始 1 次 + 重试 3 次 = 至多 4 次尝试），每次重试前 `await Future.delayed`，延迟按指数增长：1s → 2s → 4s。
   - 理由：符合需求描述的“重试 3 次 + 指数退避”，是业界常见的简单指数退避序列，无需引入随机抖动（jitter）以保持实现最小化。
   - 备选方案：使用抖动或更大基数。放弃原因：超出“改动最小”范围。

3. **循环实现方式**：在现有 `try` 块内用一个 `for (int attempt = 0; attempt <= maxRetries; attempt++)` 循环包裹尝试逻辑；单次尝试内部再用小的 `try/catch` 捕获异常并决定是否 `continue` 重试或在最后一次失败时 rethrow，交由外层原有 `catch` 统一处理失败路径（保持 `onItemDone(asset.id, null)` 逻辑不变，不重复实现）。
   - 理由：复用现有失败处理代码路径，避免逻辑重复，符合最小改动原则。
   - 备选方案：把失败处理逻辑complete复制进内层循环。放弃原因：造成重复代码。

4. **进度回调**：每次重试时，`onProgress?.call(asset.id, 0.0)` 会随着尝试重新触发（因为整个尝试单元包含从头开始的 MD5/上传流程），与需求「不改变对外回调契约」一致；不新增专门的“重试中”状态回调。

## Risks / Trade-offs

- **总耗时增加** → 多次失败重试可能使单张资源总耗时增加最多 1+2+4=7 秒的额外等待。可接受，因为只在失败路径触发。
- **重复网络请求** → 若失败发生在 `uploadFile` 后半段（服务端已收到但客户端未确认成功），重试可能造成重复上传。当前 `isExist` 秒传机制可在一定程度上降低此风险（下次尝试会先查秒传）；不做进一步处理，超出本次最小改动范围。
- **无区分错误类型** → 对不可恢复错误（如文件不存在、参数错误）也会重试 3 次，浪费时间。可接受的简化，后续可迭代为按异常类型判断。
